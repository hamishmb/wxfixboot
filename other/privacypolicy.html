I respect your privacy.

Hence, WxFixBoot collects only information it requires to run, or is of significant importance to the user, and it won't report any information it collects at all, to anybody. WxFixBoot does also not collect any personal, or personally identifiable information.

It has never done, and it never will.

Disk Information

In order to run, WxFixBoot needs to gain detailed information about disks. The information collected for this purpose is as follows:



  • Disk name (e.g. /dev/sda, /dev/cdrom).
  • Type (A partition, or a device).
  • Host Disk (e.g. /dev/sda is the host of /dev/sda2).
  • Any partitions the device contains, if any (opposite to Host Disk).
  • The Disk's Filesystem, if any (e.g. ext4, ntfs, hfs+).
  • Any flags the disk holds, which are used to get the Disk's partitioning.
  • The Disks's partitioning, if any (GPT or MBR).
  • UUID (Universally Unique Identifier), if any. These identify different disks on your system, and are generated randomly (either pseudo-randomly, or “actually” randomly).
  • ID. As above.
  • Boot Record, if any. This is used to double-check that a bootloader is present where WxFixBoot thinks it might be present.
  • LVM/RAID disks: Volume group name, and the logical volume name, used to identify the disks.

Some further Disk information is collected simply to make it easier for the user to identify their own disks. This information is as follows.



  • The Disk's Vendor (if any).
  • The Disk's Product (if any).
  • The Disk's Capacity (if any).
  • The Disk's Description (if any).

Operating System Information

A smaller amount of information needs to be collected on each Linux Operating System installed on your machine as well. This is as follows:



  • Operating System name (e.g. Ubuntu 16.04 xenial). This is used to identify each OS to the user and the WxFixBoot itself.
  • Whether this Operating System is currently running. This is used to determine how to modify this OS’s bootloader, if the user asks to do so.
  • Operating System Architecture (e.g. i386, or x86_64). This is used to determine if this OS’s bootloader can be modified later.
  • The partition the Operating System resides on. This is needed to be able to modify the OS’s bootloader if asked to by the user.
  • The Operating System’s package manager (either apt-get, or yum/dnf). As above.
  • Information from each Operating System’s /etc/fstab file, allowing WxFixBoot to:
  • Determine this OS’s EFI partition, if any (where EFI bootloaders reside and are installed to).
  • Determine this OS’s Boot partition, if any (where the OS’s Linux kernel and initrd are stored, if not on /).

Firmware Information

Your firmware type needs to be determined to check which bootloaders can be installed. For example, if you have EFI firmware, you (usually) need an EFI bootloader to start your Operating System, and installing a BIOS bootloader will not work, and vice versa. To collect this information, WxFixBoot will:



  • Check to see if the EFI variables are present. If they are, then you have EFI firmware.
  • If not, run “dmidecode -q -t BIOS” to get firmware information. If UEFI is mentioned in the output, you have UEFI firmware. If not, you have BIOS firmware.

Bootloader Information

WxFixBoot also needs to gather information about the bootloaders on your system. This information is gathered for each OS, and is as follows:



  • The currently installed bootloader (e.g. ELILO, or GRUB2).
  • Any bootloaders that are available for installation. This is used to present the user with a list of alternative bootloaders to install.
  • The disk the bootloader is installed to. If we modify this bootloader, make sure it stays installed in the same place.
  • Whether this bootloader can be modified.

Further information is gathered for your convenience, and allows the program to operate more smoothly, but is not absolutely necessary:



  • The bootloader’s timeout, if any. This allows WxFixBoot to keep it set the way it is when modifying the bootloader, and allows the user to change it as necessary.
  • The bootloader’s global kernel options, if any. As above.
  • The bootloader’s default OS, if any. As above.
  • All of the bootloader's menu entries. This is used to set and get the bootloader's default OS and match them to the OSs WxFixBoot detected.

Other Information

A basic dependency check is done to ensure that WxFixBoot can function on this system. This is worth doing because crashing while halfway through an operation is not what any user would want.

A small amount of other information is collected in order for WxFixBoot to operate correctly:



  • Whether the system is running on a live disk or not (this needs to be considered to gather OS info).
  • If so, whether this live disk is Parted Magic (this silences a few errors that would otherwise stop the program).

Finally, WxFixBoot outputs the version of itself and that of python and wxpython into the log file on startup to aid debugging if the log file is later needed to resolve a problem, for example in a support request from a user.

Conclusion

The only collected information that cannot be viewed in the GUI is the firmware type, live disk, Parted Magic information, Lvm information, /etc/fstab contents, and boot record. This information is not visible because it would be confusing to the user, and is difficult to display. If you wish to see it, instead save the log file when you exit WxFixBoot.

All the other information that is collected by WxFixBoot is accessible in the System Information Window. This window can be found in the View menu from the Main Window, or by clicking the “View More Details” button in the Bootloader Options Window.

For debugging reasons, all of this information is temporarily saved in a log file (/tmp/wxfixboot.log). It does NOT include any personal, or personally identifiable information such as serial numbers or IP addresses.

When WxFixBoot is closed, all of this information (including the logfile itself) is completely deleted, unless you decide to save it somewhere else when prompted.
